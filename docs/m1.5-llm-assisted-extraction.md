# M1.5 — Wild-Code Coverage: LLM-Assisted Extraction

## 1. One-line summary

M1.5 ships **Layer 2**: a two-stage LLM extraction pipeline that handles the 10–30% of cross-service call sites where pack rules score zero, validated at 100% precision and 100% recall on petclinic WebClient + Eureka DiscoveryClient code.

---

## 2. Why M1.5 ships as a separate milestone

Pack-only extraction is the honest M1 baseline. Spike B established that test-exclusion and producer dedup are what separates 12.5% precision from 100% — and that result needs to stand on its own before we layer probabilistic extraction on top of it. Folding LLM extraction into M1 would muddy both the precision story and the cost story.

**Spike C** ran the exact code where M1 scored 0% — Spring WebClient and Eureka DiscoveryClient in `spring-petclinic-microservices` — and returned 5/5 precision and recall using the two-stage pattern described in this doc (`spike/real-repos/SPIKE-C-FINDINGS.md`). That result is the warrant for shipping M1.5 as a real deliverable, not a research spike.

Keeping the milestones separate also means the graph at M1.5 carries legible provenance: every node is either `pack:<rule-id>` (deterministic, high confidence) or `llm:<model>@<date>` (probabilistic, medium confidence). Mixing them without that label makes the substrate opaque to downstream consumers.

---

## 3. The two-stage extraction pattern

The central insight from Spike C: **LLMs reliably extract semantic content but unreliably report line numbers**. Across two independent passes on the same file, semantic fields (method, host, path, function name) were identical; line numbers drifted by 1–3 lines and sometimes landed inside unrelated methods. One reported line for `addPetToOwner` pointed at line 51 inside `getVets` — a method with no HTTP call at all — while the real `.uri(` was at line 73.

The correct architectural response is to give each concern to the layer that handles it well.

**Stage 1 — Semantic extraction (LLM).** The model reads the file and returns a structured JSON array. Each record contains `method`, `host`, `path`, `function`, and `line`. The `line` field is accepted but treated as a hint only — it is never persisted directly to the graph. The model's job is to understand framework semantics: resolve a `discoveryClient.getInstances("customers-service")` chain to the host `customers-service`, normalise `/owners/{ownerId}` from a URI template call, and identify the enclosing function name.

The prompt instruction is tightly scoped: extract HTTP client calls (WebClient, RestClient, RestTemplate, or equivalent), one record per call site, host must be the downstream service name, path must omit the host and use `{curlyBraces}` for path variables. "Do not invent calls that are not in the code" is explicit in the instruction.

**Stage 2 — Anchor relocation (deterministic).** For each claim that survived two-pass consensus, a deterministic post-processor locates the real source anchor. It does not use the LLM-reported line. The algorithm is in section 5. If no anchor is found, the claim is **dropped** — better to lose one edge than to emit a node pointing at the wrong line in the wrong method.

---

## 4. Two-pass consensus protocol

**Both passes must agree** for a claim to be emitted. A claim from pass 1 is matched against pass 2 claims using the semantic key:

```
(method, host, normalised_path, function_name)
```

All four fields must match exactly. `normalised_path` is the path after stripping the query string, ensuring a leading `/`, removing any trailing `/`, and stripping a leaked host prefix if the model included one. HTTP method comparison is case-insensitive; host and function comparisons are exact.

A claim present in pass 1 but absent in pass 2 — or vice versa — is discarded. A false positive that survives both passes independently would require the model to hallucinate the same non-existent call twice with identical semantic fields. In practice, Spike C produced zero such cases across all three files.

Line numbers are explicitly excluded from the consensus key. The goal is to filter hallucinated edges, not to require line-number agreement (which the model cannot reliably provide).

Two-pass cost is `2 × $0.00015 = $0.0003` per file — sub-cent for any realistic file size.

---

## 5. Anchor relocation algorithm

Anchor relocation is **the architecturally critical part of M1.5**. Without it, line numbers in the graph are wrong — sometimes by 13 lines, sometimes pointing at an entirely different method. The algorithm trusts the function name and host literal, not the LLM's line estimate.

For each consensus claim `(method, host, path, function_name)`:

1. **Find the method declaration.** Scan the file line by line for a non-comment line containing ` <function_name>(` or starting with `<function_name>(`. Record that line as `method_start`.

2. **Scan the method body.** Take the next 30 lines from `method_start`. For each line in that window:
   - Check if the host string appears literally (`host in line`).
   - Check if a DiscoveryClient lookup appears: `getInstances("<host>")`.
   - If either is true, check that the surrounding ±3-line window contains an HTTP-shape token: `.uri(`, `.retrieve()`, `RestClient`, `WebClient`, or `RestTemplate`.
   - If the HTTP token check passes, return `method_start + offset` as the verified anchor line.

3. **Handle field-stored hostnames.** If the host literal is not inside the method body (because it is stored in a class field, as in `VisitsServiceClient`), the body scan falls back to scanning class-level field declarations for a string assignment containing the host, then confirming that the call site references that field name within an HTTP-shape expression. This is the one miss from Spike C (1/5 anchor misses) and is a required M1.5 deliverable. For Java/Kotlin, scan `private String <field> = "...<host>..."` declarations; for JS/TS, scan module-level `const` assignments.

4. **Confirm or discard.** If a verified anchor is found, emit the graph node at that line with `confidence: medium`, `source: llm:<model>@<date>`. If no anchor is found, **drop the claim and log the miss** — do not emit the node.

The reference implementation of steps 1–3 is in `spike/real-repos/probe_llm_extract.py`, `locate_anchor()`. The field-inlining extension in step 3 is not yet in that file and is a required addition for M1.5.

## 6. Provenance and confidence

Every node emitted via the LLM path carries two mandatory properties:

- `source: llm:<model>@<date>` — the specific model identifier and ISO date of the extraction run. Example: `llm:gpt-4o-2024-08-06@2025-01-15`. This is distinct from `pack:<rule-id>` nodes produced by Layer 1.
- `confidence: medium` — default for all LLM-extracted nodes. Pack-extracted nodes carry `confidence: high`. Agent-extracted nodes (Layer 3, M2) carry `confidence: low` by default unless they confirm a pack or LLM claim.

**Provenance is queryable.** The graph schema must support filtering by `source` prefix and `confidence`. Consumers that only want deterministic results filter to `source STARTS WITH "pack:"`. When a pack rule is later authored for a pattern that was LLM-extracted, the re-index overwrites the `llm:` node with a `pack:` node at `confidence: high`.

---

## 7. Coverage-aware triggering

**Only files below a coverage threshold are sent to the LLM.** After Layer 1 pack extraction runs on a file, the engine computes a coverage score: the ratio of function-body lines that contain at least one outbound-edge candidate (endpoint call, message publish) matched by a pack rule. If that ratio is above threshold (default: 0.8), the file is considered pack-covered and the LLM is not invoked.

The threshold is configurable per language pack. A Spring-heavy repo with mature WebMvc and RestTemplate packs will rarely trigger LLM extraction. A repo using a bespoke HTTP wrapper or a third-party SDK with no pack will trigger it for nearly every file containing HTTP calls.

**Coverage-aware triggering** keeps steady-state cost negligible. In a stable polyrepo, only new or changed files fall below threshold on a given re-index. The caching layer in section 8 reduces cost to near-zero for everything else.

---

## 8. Caching by `(content_hash, prompt_version, model_version)`

Every LLM extraction result is cached keyed on the tuple:

```
(sha256(file_content), prompt_version, model_version)
```

All three components are mandatory. A cache hit returns the stored JSON and skips the model call entirely. A cache miss invokes the model, stores the result, and records the cost.

**Model-version pinning is required.** The `model_version` in the cache key must be the full, specific model string (e.g., `gpt-4o-2024-08-06`), not an alias like `gpt-4o`. Aliases resolve to different model weights over time, which causes silent semantic drift without a cache miss. When the team intentionally upgrades the model, bump `model_version` in config — this forces a full re-index at the next run and rebuilds the cache.

Similarly, `prompt_version` must be bumped whenever the instruction text or the output schema changes. A semver string is sufficient (`v1`, `v2`).

With this cache in place, the $1.20 full re-index cost from Spike C is a one-time number. Subsequent re-indexes on a stable polyrepo cost roughly $0 for unchanged files and ~$0.0003 per changed file.

## 9. Structured-output schema requirements

The LLM is called with a strict JSON schema enforced by the structured-output endpoint. The schema must be:

```json
{
  "type": "object",
  "additionalProperties": false,
  "required": ["calls"],
  "properties": {
    "calls": {
      "type": "array",
      "items": {
        "type": "object",
        "additionalProperties": false,
        "required": ["method", "host", "path", "function", "line"],
        "properties": {
          "method":   { "type": "string" },
          "host":     { "type": "string" },
          "path":     { "type": "string" },
          "function": { "type": "string" },
          "line":     { "type": "integer" }
        }
      }
    }
  }
}
```

`additionalProperties: false` must appear at both the top-level object and each item object. Spike C found that omitting it caused the model to silently return empty arrays — the structured-output endpoint treated the partial response as valid and swallowed the calls. Strict schema enforcement is non-negotiable.

**Validation is mandatory on every response.** Before the post-processor runs, validate the raw JSON against the schema. On validation failure:

- Log the full response and the validation error at `ERROR` level.
- Do not emit any nodes for this file.
- Increment a `llm_schema_validation_error` metric counter.
- Do not silently continue — schema failures indicate a prompt regression or model change and must surface in monitoring.

The prompt instruction and schema are coupled. Changes to either require bumping `prompt_version` and re-running the CI benchmark (see section 14).

## 10. Failure modes and mitigations

**Model-version drift.** When the model behind an alias changes, extraction output can shift without a cache miss. Mitigation: always pin the full model string, never use an alias. On intentional model upgrade, bump `model_version` in config and schedule a full re-index; surface the new `source:` values in the graph so operators can diff before and after.

**Schema drift.** Changes to the output schema without bumping `prompt_version` will produce cache hits on old format data. Mitigation: treat `prompt_version` as a required field in CI and gate deploys on a passing schema-validation test against the current schema and current model.

**Long files exceeding context.** Spike C files were under 120 LOC and fit comfortably in a single prompt. Production files may exceed context limits. Mitigation: chunk at method boundaries before sending. For Java/Kotlin, split on method declaration lines; send each method as an independent extraction request, tagged with the enclosing class name. For JS/TS, split at export-level function or arrow-function boundaries. The anchor relocation algorithm is unchanged — it still operates on the full original file.

**Field-stored hostnames.** The Eureka `VisitsServiceClient` pattern — host in a `private String` field, concatenated at call time — caused the one anchor miss in Spike C (4/5 instead of 5/5). The semantic extraction was correct; only anchor relocation failed. Mitigation: extend the anchor locator to scan class-level field declarations for the host string when the method-body scan fails, then confirm the field name appears in an HTTP-shape expression in the method body.

**DiscoveryClient and other resolution indirections.** `AIDataProvider` has no URL literal at all — the host comes from `discoveryClient.getInstances("customers-service").get(0).getUri()`. The LLM handled this correctly in Spike C. The anchor locator handles it via the `getInstances("<host>")` pattern check. New resolution patterns (e.g., service mesh sidecars, custom registry clients) will require extending both the prompt instruction and the anchor locator — but because LLM extraction is language-semantic rather than AST-pattern-based, the extraction step often handles novel patterns without prompt changes. Only the anchor locator needs code changes.

## 11. Promotion path

The long-game story for Layer 2 is that **the probabilistic layer feeds the deterministic layer**. LLM-extracted nodes are not a permanent cost centre — they are signal waiting to be crystallised into pack rules.

The promotion path operates on a quarterly cadence:

1. A batch job clusters all `llm:`-sourced nodes in the graph by AST shape. "AST shape" means the structural pattern of the call site: framework class, chained method names, argument count and types. Two call sites with the same shape in different repos are candidates for the same pack rule.

2. Clusters with occurrence count above a threshold (proposed: 10+ instances across 3+ repos) are surfaced as **candidate pack rules** in the auto-author queue. Each candidate shows the AST pattern, the repos it was seen in, the LLM-extracted semantic fields, and a draft YAML rule for human review.

3. A human engineer reviews the draft in the auto-author UI (introduced in M2), edits if necessary, and merges the rule to the pack library. On the next re-index of any affected repo, the call sites that matched the new pack rule are re-extracted as `pack:<rule-id>` nodes at `confidence: high`, and the `llm:` nodes for those sites are retired.

Over time, the repos with the highest LLM extraction rates become pack-covered. The result is a self-improving system: the LLM handles the long tail on day one, and the pack library expands to cover it durably over subsequent quarters without any manual audit of individual files.

## 12. Cost model

**Concrete numbers from Spike C:**

| Unit | Cost |
|---|---|
| Single LLM extraction call, ~50 LOC Java file | ~$0.00015 |
| Two-pass extraction (both passes), same file | ~$0.0003 |
| 200-file Spring service, full LLM-assisted index | ~$0.06 |
| 20-service polyrepo (~4,000 files), full re-index | ~$1.20 |
| Incremental re-index, 10 changed files | ~$0.003 |

"Full re-index" means sending every file below coverage threshold to the LLM — a worst-case number for a repo with no packs. A mature Spring repo with good pack coverage will send far fewer files.

For a team running daily re-indexes on a stable polyrepo, total monthly LLM cost is realistically under $5.

These numbers reflect model pricing at Spike C's run date; re-validate when `model_version` is bumped.

## 13. Out of scope

**Query-time agents (Layer 3)** are M2. Layer 3 invokes an LLM at query time to walk gaps in the graph when a user asks a specific question — for example, tracing a flow through a service where neither packs nor Layer 2 extraction have produced coverage. That is a different cost model, a different latency budget, and a different triggering condition from the index-time batch extraction in this doc.

M1.5 does not include MCP or query-time tooling (M2), the web UI (M3), polyglot support beyond Java/Kotlin/JS/TS (M2 add-on), gRPC or service mesh call extraction (M2), or the auto-author UI (M2). The quarterly batch job and candidate data structure that feed that queue are M1.5.

## 14. Success metrics

**The Spike C result must be reproducible at integration test time.** The CI suite for M1.5 must include a benchmark that:

- Runs Layer 2 extraction on the three petclinic files used in Spike C (`CustomersServiceClient.java`, `VisitsServiceClient.java`, `AIDataProvider.java`)
- Asserts 5/5 recall (all five ground-truth edges recovered)
- Asserts 5/5 precision (no false-positive edges in consensus output)
- Asserts ≥4/5 anchor-verifiable (at least 4 of 5 claims locate to a verified line; the fifth requires field-inlining and must be 5/5 once the field-scan extension lands)

Secondary metrics:

- Schema validation error rate must be 0% on known-good test files
- Cache hit rate must be 100% on second run of unchanged files
- LLM triggering rate on a pack-covered repo (e.g., `piomin/sample-spring-kafka-microservices` after Spike B pack tuning) must be below 10% of files — confirming coverage-aware triggering is working

The benchmark is the executable contract between M1.5 and M2. M2's MCP layer can assume 5/5 edge coverage on petclinic WebClient + DiscoveryClient code.

## 15. Open questions

**1. Field-inlining scope for anchor relocation.** The current algorithm handles `private String field = "http://host/"` in Java. JS/TS module-level `const` assignments are the obvious next case, but React and Node patterns vary enough that a generic solution may be non-trivial. Owner: whoever implements the anchor relocator. Decision needed before M1.5 is marked complete.

**2. Chunking strategy for files over context limit.** Per-method chunking is the proposed approach, but it requires an accurate method-boundary detector before the LLM call. For Java this is tractable with a simple brace-counter; for Python the indentation rules are different. Should the chunker use tree-sitter for boundary detection, or is a heuristic sufficient? Decision needed before enabling Python support in M1.5.

**3. Promotion threshold calibration.** The proposed cluster threshold (10+ instances across 3+ repos) is a guess. Too low and the auto-author queue fills with one-off patterns; too high and durable patterns stay probabilistic for too long. Needs empirical tuning once M1.5 has run on real polyrepos for a quarter. Owner: whoever owns the auto-author queue.
